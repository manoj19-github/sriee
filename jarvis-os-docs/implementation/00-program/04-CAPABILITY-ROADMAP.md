# Capability Delivery Program

Source: [Sriee and JARVIS capability roadmap](../../41-SRIEE_CAPABILITY_ROADMAP.md).

## Purpose

This program turns the broad capability catalogue into an ordered implementation sequence. It does not override the canonical function maps or stage gates.

## Phase A — Safe task core

Deliver authenticated configuration/lifecycle, durable tasks/events, cancellation, approvals, replay, LangGraph state, policy, audit, PostgreSQL migrations and a fake Windows executor.

Exit: restart, duplicate, cancel, expired approval, forged approval and partial-failure tests pass.

## Phase B — Developer coworker vertical slice

Deliver registered projects, scoped files/processes, VS Code, build/test, Git and Docker actions. Add typed receipts, postcondition verification and project-root confinement.

Exit: “continue my project” succeeds in clean, dirty, offline and recoverable-failure Windows VM scenarios.

## Phase C — Daily organization

Add task/reminder scheduling, calendar, weather, GitHub and notification adapters; morning/evening briefings; goals, focus/Pomodoro and quiet hours.

Exit: timezone/DST, stale connector, duplicate reminder, missed notification, private lock-screen and revoke tests pass.

Add the developer-coach profile, learning plans, trusted progress, lessons/quizzes/challenges, project/career coaching, achievements and bounded proactive workflow suggestions through `220000–220015`.

Exit: fabricated progress, copied protected learning content, no-consent inspection, reminder fatigue, inappropriate humor and external career-profile write tests pass.

## Phase D — Sriee conversation

1. Ship text-only Sriee profile and relationship boundary filter.
2. Add female TTS and interruption.
3. Add contextual multi-turn conversation and approved memory.
4. Add original stories and jokes.
5. Add opt-in romantic mode, preferred terms and flirt intensity.
6. Add immediate romance stop, serious-context suppression and private-mode controls.

Exit: affection/romance opt-in, rejection, intensity, topic exclusion, age safety, copyrighted text, privacy, coercion, dependency, jealousy, isolation, approval influence and memory-misuse evaluations pass.

## Phase E — Voice and visible sensors

Add microphone sessions, wake word, local/online STT, camera sessions, presence, optional local face enrollment/matching, expression cues, QR/document scanning and screen understanding.

Exit: indicators, kill/revoke, false wake, uncertain transcript, device loss, raw-artifact expiry, liveness/spoof, false match, deletion, cloud-egress denial and forbidden-inference tests pass.

Optional general-wellness records and connectors (`220016–220024`) follow separate consent, storage, egress and deletion controls. Ship manual entry before provider connections and descriptive display before proactive reminders.

Exit: source/unit/freshness, conflict, medication-boundary, diagnostic-claim, urgent-language, lock-screen privacy and verified deletion tests pass.

## Phase F — Desktop, media and information

Add typed adapters for supported Windows settings, file organization, browser research, YouTube/media playback, music providers, dashboard widgets and system-health observations.

Exit: capability/resource scoping, platform-rule compliance, transfer/form approval, destructive preview, secret-field exclusion and adapter rollback tests pass.

## Phase G — Communications and research

Add email, calendar writes, meetings, research sources and unified notifications. Draft/read paths ship before send/write paths.

Exit: recipient/attendee/content preview, prompt injection, malicious attachment, duplicate send, revoke and audit tests pass.

## Phase H — Extensibility and future devices

Add signed out-of-process plugins before marketplace integrations. Smart home, mobile, wearable, synchronization, multi-user and enterprise features receive separate identity, privacy and threat models.

Exit: signature, isolation, quota, permission-diff, update, revocation, uninstall/data removal and compromised-plugin tests pass.

## Cross-cutting rule

Offline/local processing is preferred for sensitive audio, video, biometric and memory data. Cloud processing is provider-neutral, classified, purpose-bound and disabled without an eligible egress grant.

No phase introduces a blanket “full control” authority. Sriee and every specialist submit the same typed actions to the same policy, approval, executor, verification and audit path.
